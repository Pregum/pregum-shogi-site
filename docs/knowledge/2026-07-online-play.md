# 対人戦（オンライン対局）の仕組み（2026-07）

対象: `src/worker/index.ts`（Worker本体）、`src/worker/room.ts`（ShogiRoom Durable Object）、`src/shared/protocol.ts`（メッセージ定義）、`src/client/lib/useRoom.ts`（クライアント側の接続フック）

## 1. 全体アーキテクチャ

```
ブラウザA ─┐
            ├─ WebSocket ─→ Cloudflare Worker ─→ ShogiRoom (Durable Object)
ブラウザB ─┘                 (ルーティングのみ)      1ルーム = 1オブジェクト
```

- **1ルーム = 1 Durable Object（DO）**。`env.ROOM.idFromName(roomId)` でルームIDから常に同じDOインスタンスに到達する。DOは「そのルームの状態を持つ小さなサーバー」で、同時アクセスが直列化されるため排他制御が不要
- Workerは `POST /api/rooms`（ルームID発行）と `/api/rooms/:id/ws`（WebSocketをDOへ転送）の2エンドポイントだけ。静的アセット(HTML/JS/CSS)は wrangler の assets 設定が配信する
- ルームIDは `狐関連の単語-ランダム4文字`（例: `kitsune-d4qf`）。**URLを知っていることが入室権限**というシンプルなモデル

## 2. 座席とトークン（誰が先手/後手か）

- クライアントは初回アクセス時に `crypto.randomUUID()` でトークンを生成し localStorage に保存（`storage.ts` の `getToken`）
- `join` メッセージでトークンを送ると、**先着2名（異なるトークン）が先手・後手の座席を取得**。3人目以降は観戦者
- 再接続時は同じトークンを送るので同じ座席に戻れる（リロード・回線切断に強い）
- トークンは他のクライアントには**一切送信されない**（`snapshot()` は名前とオンライン状態だけを含む）ため、座席の乗っ取りはできない

## 3. メッセージプロトコル（`protocol.ts`）

クライアント→サーバー（`ClientMsg`）:

| type | 内容 |
| --- | --- |
| `join` | トークン・名前・（部屋作成者のみ）持ち時間 |
| `move` | 指し手 |
| `matta` | 待ったの request / accept / reject |
| `rematch` | 再戦の request / accept / reject |
| `resign` | 投了 |

サーバー→クライアント（`ServerMsg`）は実質 `state` の1種類のみ。**何かが起きるたびに全接続へ最新の `GameSnapshot`（棋譜全体・両者情報・時計・結果）をブロードキャスト**する。差分配信ではなく全量配信にすることで、クライアントは「受け取ったスナップショットを描画するだけ」になり、状態のズレ・順序問題が原理的に起きない（棋譜は高々数百手なのでサイズも問題ない）。

## 4. チート対策: サーバー側検証

指し手はサーバー（DO）側で必ず検証する（`room.ts` の `move` ハンドラ）:

1. 手番の照合（あなたの番か？）
2. `legalMoves(pos)` を生成し、送られてきた手と `moveEquals` で照合
3. 一致した**サーバー側の合法手オブジェクトを保存**（クライアントの送信値をそのまま信じない）

ルールエンジン `src/shared/shogi.ts` をクライアントと共用しているので、クライアントは合法手のハイライト表示に、サーバーは検証に、同じコードを使う（二重実装なし）。

## 5. 待った・再戦の承認フロー

`pending: { kind: 'matta' | 'rematch', by: Color }` を1つだけ持てる状態機械:

- **待った**: 申請 → 相手に承認モーダル表示 → accept で「直近の自分の手まで」巻き戻し（相手が指した後なら2手、直後なら1手）。reject で何もしない
- **再戦**: 終局後に申請 → accept で棋譜クリア・**先後入替**・`gameNo` インクリメント・時計リセット

## 6. 持ち時間（対局時計）

- 部屋の作成者（最初のjoin）が `timeControl`（分）を指定。両者の残り時間(ms)をDOが管理
- 着手時に「手番開始からの経過時間」を残り時間から差し引く
- **時間切れの自動判定はDOのアラーム**（`ctx.storage.setAlarm`）で行う。手番側の残り時間が尽きる時刻にアラームを仕掛け、発火時に `alarm()` が時間切れ負けを確定してブロードキャストする。**両者が画面を閉じていても時間切れが成立する**のがポイント
- クライアントの時計表示はスナップショットの `serverNow`（サーバー時刻）を基準に補正するので、端末の時計がズレていても表示が狂わない

## 7. 千日手

着手のたびに `checkRepetition(positions)` で判定（`shogi.ts`）:

- 局面キー = 盤面 + 持ち駒 + 手番 の文字列。最終局面と同一のキーが**4回**出現したら千日手
- その間の一方の手が**すべて王手**なら「連続王手の千日手」で王手側の負け、そうでなければ引き分け

## 8. 接続管理

- **WebSocket Hibernation API**（`ctx.acceptWebSocket`）を使用。接続がアイドルのときDOはメモリから退避でき、コストがかからない。トークンは `serializeAttachment` でソケットに紐付けて休止をまたいで保持
- クライアント（`useRoom.ts`）は切断時に**指数バックオフ（0.5s→8s上限）で自動再接続**し、再joinする
- オンライン表示: ブロードキャスト時に接続中ソケットの持ち主を調べて `online: [bool, bool]` を含める

## 9. 永続化

- ゲーム状態はDOの `ctx.storage`（SQLiteバックエンド）に毎操作後 `put`。DOが再起動しても対局は消えない
- `new_sqlite_classes` を使っているため **Cloudflare無料プランで動く**（`wrangler.jsonc` の migrations 参照）
- 棋譜の長期保存はサーバーではなく**クライアントの localStorage**（終局時に自動保存）。サーバー側にユーザー概念を持たないための設計判断
